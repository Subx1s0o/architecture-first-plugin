---
description: Authorize + execute a cleanup batch. Writes ephemeral /tmp marker to lift mass-deletion gate for one batch. Optional --pr flag opens a draft PR automatically. --auto skips interactive confirmation (CI mode).
---

Takes `<batch-id>` (e.g. `CLN-003-batch-B`) [flags].

Flags: `--pr` (create branch, commit, push, open draft PR), `--no-pr` (default, commit on current branch only), `--label <name>` (default `arch-cleaner`), `--base <branch>` (default `.arch-profile.yaml` `default-branch:` or `main`), `--auto` (skip the interactive "yes/approve" gate at step 3 — for CI / autonomous runs where there is no human to answer).

1. Locate the manifest file under `docs/cleanup/`. Read the batch section.
2. Refuse if the batch contains any L3/L4 finding — those need architect review first. Emit a clear error pointing at `/arch-review`.
3. List every file and line-range the batch will delete, with the per-row evidence from the manifest. **If `--auto`:** print the list and proceed without asking. **Otherwise:** ask the user for explicit confirmation ("yes" / "approve"). If the user declines, stop.
4. Write the cleanup marker at `${TMPDIR:-/tmp}/architecture-first/<md5(CLAUDE_PROJECT_DIR)>-<session_id>.clean-approved` with `batch: <batch-id>` and an ISO timestamp. Create the parent dir if missing. Do not write anywhere under `<project>/.claude/`.
5. **If `--pr`:** create branch `chore/cleanup-<batch-id>` from current HEAD before applying edits. Otherwise stay on current branch.
6. Execute the deletions as one logical change:
   - L1 formatting/logs: apply edits.
   - L2 orphan files: `git rm` (or `rm` if not tracked).
   - L2 deps: edit the stack manifest (`package.json`, `pyproject.toml`, `go.mod`, …) and regenerate the lockfile per the detected stack.
7. Run the stack's test and build commands from `.arch-profile.yaml` `commands:`. If anything fails, `git restore` and report the failure. **If `--pr`** and the branch was created in step 5, also `git checkout <base>` and `git branch -D <branch>` before reporting.
8. Commit the deletions: `chore(cleanup): <batch-id> deletions`. Sanity check before staging: if the batch's manifest path itself (`docs/cleanup/CLN-*.md`) shows up in `git status`, the manifest must NOT be staged in this commit — the plugin's manifest writes are documentation, not part of the cleanup diff.
9. **If `--pr`:** push and open draft PR. Otherwise skip to step 10.

   ```bash
   cd "$REPO_ROOT" && git push -u origin "$BRANCH"
   ```

   On push success:

   ```bash
   LABEL="${LABEL:-arch-cleaner}"
   PR_URL=$(gh pr create \
     --draft \
     --base "$BASE" \
     --head "$BRANCH" \
     --label "$LABEL" \
     --title "chore(cleanup): <batch-id>" \
     --body "$PR_BODY" 2>&1) || PR_OPEN_FAILED=1
   ```

   PR body shape:

   ```
   ## Summary

   Auto-generated cleanup PR from architecture-first plugin (<batch-id>).

   <one-paragraph from CLN manifest header>

   ## What this PR does

   <full bullet list copied from the executed batch section of the CLN manifest — file path, line range, kind, one-line reason for each entry>

   ## Safety guarantees

   - Only the entries listed above (no L3/L4 findings)
   - `<commands.test>` passed in CI before push
   - `<commands.build>` passed in CI before push

   ## How to review

   Skim the diff. Each deletion should be obviously dead. If anything looks live or surprising, **do not merge** — close the PR.

   ---

   Generated by `architecture-first` plugin. Marked **draft** for human review.
   ```

   Failure handling: same as `/arch-execute` step 7.5 — never abort the run, print the manual `gh pr create` command, continue.

   On success: print `🔗 PR opened: <PR_URL>`. If `GITHUB_STEP_SUMMARY` is set, append PR URL there.

10. Append the manifest with `Status: executed on <date>, commit <sha>[, branch <branch>][, pr <PR_URL>]` and the test/build result.

The cleanup marker is session-scoped and ephemeral; it does not authorize any subsequent batch.
